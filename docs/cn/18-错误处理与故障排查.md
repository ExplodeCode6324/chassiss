# 错误处理与故障排查

中文 | [English](../en/18-troubleshooting.md)

CHASSISS 错误包含稳定 code、退出码、retryable 标记、remediation 和可选 diagnostic category。自动化应读取 `--json` 结果。人类可以使用 `explain` 查看内置说明。

```sh
chassiss explain CHS-CONFLICT-REVISION
```

## 处理顺序

遇到拒绝时按以下顺序检查。

1. 停止当前写操作
2. 保存完整 JSON 错误
3. 运行 `bootstrap`
4. 按 `context_requests` 读取目标资源
5. 若要求恢复，先运行 `recover`
6. 运行 `doctor` 与 `verify`
7. 依据新 revision 重新判断动作

完整性错误出现后不要修改 `.chassis` 或 Git refs。

## 退出码分类

| 退出码 | 常见含义 |
| --- | --- |
| 1 | 未分类内部错误 |
| 10 | 当前资源状态不允许该动作 |
| 11 | 身份、权限或 scope 拒绝 |
| 12 | revision 或 Git 并发冲突 |
| 20 | 命令参数或输入格式错误 |
| 30 | 外部系统暂时失败 |
| 40 | 数据或协议完整性异常 |

应以错误码为主，退出码用于粗粒度分类。

## 认证失败

| 错误码 | 检查内容 |
| --- | --- |
| `CHS-AUTH-MISSING` | 是否提供了 `--credential` |
| `CHS-AUTH-CREDENTIAL` | credential 项目、metadata、policy 和 key 是否匹配 |
| `CHS-AUTH-REVOKED` | grant 是否已被 Master 回收 |
| `CHS-AUTH-NOT-YET-VALID` | `not_before` 是否尚未到达 |
| `CHS-AUTH-EXPIRED` | credential 是否过期 |
| `CHS-AUTH-DENIED` | role 和 action scope 是否允许命令 |
| `CHS-AUTH-RESOURCE` | Project、Mission、Task、Submission 或 commit scope 是否覆盖目标 |

不要通过编辑 credential 延长有效期或扩大 scope。Master 应签发新的最小权限 credential。

`diagnostic_category` 可以进一步区分 `grant_not_found`、`revoked`、`metadata_mismatch`、`policy_mismatch`、`key_invalid`、`key_mismatch`、`signature_invalid` 和 `project_mismatch`。

## revision 冲突

`CHS-CONFLICT-REVISION` 表示 State 已在读取后更新。`CHS-CONFLICT-TRUST-REVISION` 表示授权集合已变化。

处理方式如下。

```text
重新 bootstrap
读取目标 context
确认原动作仍然需要
使用新 revision 执行
```

revision 冲突本身通常可以重试，业务动作不能盲目重复。

## 项目锁

`CHS-CONFLICT-LOCKED` 表示另一个进程持有内核级 advisory lock。

等待当前命令结束后重试。不要按 `.chassis/lock` 的修改时间判断锁是否陈旧，也不要删除锁文件。

## 范围与预算

| 错误码 | 处理 |
| --- | --- |
| `CHS-WORK-SCOPE` | 将范围外修改移入新的受控 Task |
| `CHS-WORK-BUDGET-FILES` | 按文件边界拆分 Task |
| `CHS-WORK-BUDGET-LINES` | 缩小交付或提交替代 Task |
| `CHS-WORK-BUDGET-COMMITS` | 创建干净的替代 Task 分支 |
| `CHS-TASK-PATH-CONFLICT` | 等待先行 Task 或重新划分路径 |
| `CHS-TASK-WIP` | 完成、释放或阻断已有 Task |

冻结 Task 的预算不能在执行中直接放宽。Designer 提交新 Task，Master 接受后再由 Orchestrator supersede。

## Check 失败

先读取具体 Check ID、退出码和输出末尾。随后在 Task worktree 中检查以下项目。

- 当前 Head 是否已提交
- `cwd` 是否存在且位于项目内
- 命令依赖是否已安装
- timeout 是否合理
- 必需环境变量是否在 CheckSpec 中声明
- verification source 是否变化
- 检查结果是否对应当前 snapshot

修复后重新运行 `work check`。Reviewer 和 Integration 失败时，需要回到 Developer 流程产生新 Submission。

## worktree 异常

移动 worktree、切换分支或手工清理可能触发 binding 错误。

1. 停止在该目录继续提交
2. 运行 `work status <task-id>`
3. 运行 `doctor` 与 `verify`
4. 存在 journal 时运行 `recover`
5. 将诊断交给 Orchestrator 或 Master

不要新建同名目录伪装原 worktree。

## Review 与 Integration

| 错误码 | 处理 |
| --- | --- |
| `CHS-REVIEW-INDEPENDENCE` | 使用不同 actor 的 Reviewer |
| `CHS-INTEGRATION-NOT-APPROVED` | 检查 Task 和 Submission verdict |
| `CHS-INTEGRATION-REVIEW` | 由作出批准的 Reviewer 执行 |
| `CHS-INTEGRATION-HEAD-MOVED` | 对新 Head 重新 submit 和 review |
| `CHS-INTEGRATION-BASELINE-MOVED` | 运行 verify，禁止 force |
| `CHS-INTEGRATION-CONFLICT` | 在受控 Task 流程中协调冲突 |
| `CHS-INTEGRATION-CHECKS` | 修复后重新产生 Submission |

## Publish

| 错误码 | 处理 |
| --- | --- |
| `CHS-PUBLISH-DIRTY` | 清理或受控提交正式工作区变化 |
| `CHS-PUBLISH-BASELINE` | 检查默认分支与 State baseline |
| `CHS-PUBLISH-NON-FAST-FORWARD` | 将远端变更通过 Task 纳入本地 |
| `CHS-PUBLISH-PUSH` | 先 recover，再判断重试 |
| `CHS-PUBLISH-REMOTE-UNSAFE` | 使用不含外部 helper 或控制字符的 URL |
| `CHS-PUBLISH-REMOTE-MISMATCH` | 停止发布并检查远端 |

Publish 永不 force push。

## integrity 错误

`CHS-INTEGRITY-EVENTS`、`CHS-INTEGRITY-TRUST`、`CHS-STATE-INVALID` 和 `CHS-INTEGRITY-BLOCKED` 表示控制证据无法通过验证。

保留以下材料。

- 完整错误 JSON
- `.chassis` 的只读备份
- `git status`、refs 和 reflog
- `doctor --json` 与 `verify --json`
- 相关 journal ID
- 最近一次成功命令和时间

不要删除 journal、重写事件、覆盖 Trust 或 reset 正式分支。由 Master 决定后续恢复。

## 旧 schema

`CHS-SCHEMA-UNSUPPORTED` 表示项目不符合当前 API V2 和 schema 版本。当前版本没有原地迁移器。保留旧项目，使用兼容 CLI 读取，或在评估后初始化新的 V2 控制端。

## 提交 Issue

报告问题时删除 credential、Root、token、远端 URL 和私有源码。保留版本号、平台、命令形式、错误码、最小复现、预期结果、实际结果和是否存在 journal。
