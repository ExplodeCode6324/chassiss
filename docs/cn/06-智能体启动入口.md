# 智能体的 `bootstrap` 入口

中文 | [English](../en/06-bootstrap.md)

`bootstrap` 验证 credential，并返回该身份在当前项目状态下可以使用的能力、上下文和候选动作。

```text
chassiss --json \
  --root /path/to/project \
  --credential /path/to/credential.yaml \
  bootstrap
```

Agent 每次启动都应先执行该命令。

## 返回内容

Bootstrap schema 当前为 `chassiss.bootstrap/v3`。

| 字段 | 用途 |
| --- | --- |
| `schema_version` | Bootstrap envelope 版本 |
| `binary_version` | CLI 版本 |
| `project_root` | 解析后的项目根目录 |
| `state_revision` | 当前工作流 revision |
| `trust_revision` | 当前授权 revision |
| `principal` | credential 推导出的 actor、role、actions 和 scope |
| `policy` | Role Policy version、digest 和公共约束 |
| `capabilities` | 该 credential 可调用的命令 schema |
| `available_actions` | 当前 State 下的候选 mutation |
| `context_requests` | 执行动作前应读取的上下文 |
| `refresh_on` | 需要重新 bootstrap 的条件 |

## principal

`principal` 直接来自验证后的 credential 与 trust grant。

- `credential_id`
- `actor`
- `role`
- `actions`
- `resources`
- `not_before`
- `expires_at`
- `persistent`

Agent 不能在 `bootstrap` 中传入 role。CLI 会拒绝自报身份。

## policy

Role Policy 当前版本为 3。`policy.digest` 固定命令、角色映射和公共约束。策略有意变化时需要提升版本或更新 golden digest。

公共约束包括以下内容。

- 可信 CLI、签名事件链、当前 trust revision 和 State 共同构成权威
- Agent 不编辑 `.chassis/`
- `available_actions` 只表示当前 revision 的候选动作
- 每条命令会重新验证授权和状态
- 冲突、拒绝、trust 变化和 credential 轮换后重新 bootstrap

## capabilities

`capabilities` 给出当前 credential 的命令集合。每项包含 usage、summary、参数、flag 和 mutation 属性。

Agent 只使用这里出现的命令。文档示例可能落后于二进制，capability schema 保持机器可读。

## available_actions

候选动作由 role、actor、credential actions、资源 scope、State 和 trust 共同过滤。

一个 action 包含以下信息。

- action 名称
- argv
- resource
- reason
- required inputs
- optional inputs
- optional 标记

例如 Developer 的 `work.check` 会直接带上 `--all`。`work.submit` 会列出必须提供的 handoff 与可选 message。

候选动作不会跳过命令级验证。另一个 Agent 可能先推进 revision，credential 也可能在执行前被回收。

## context_requests

CLI 根据角色返回应读取的上下文。

| 角色 | 常见 context |
| --- | --- |
| Designer | 被拒 Artifact 的 `artifact context` |
| Orchestrator | 当前 Mission 的 `mission context` |
| Developer | 已分派 Task 的 `work context` |
| Reviewer | 待复核 submission 的 `review context` |
| Master | 待接受 Artifact 与活动 Mission |

Agent 在修改资源或给出 verdict 前执行这些 argv。

## revision CAS

Mutation 应传入最近一次 Bootstrap 返回的 `state_revision`。

```text
chassiss --expect-revision 12 \
  --credential /path/to/credential.yaml \
  work check M001-T001 --all
```

revision 已变化时，CLI 返回 `CHS-CONFLICT-REVISION`。该错误可重试，但调用方必须先重新 bootstrap 并重新评估动作。

授权 mutation 使用 `--expect-trust-revision`。

## 重新 Bootstrap 的条件

至少在以下情况重新执行。

- mutation 成功
- revision conflict
- CLI rejection
- Artifact rejection
- trust revision 变化
- credential 轮换或回收
- 选择新的 Mission、Task 或 submission
- Session 被重新唤醒

长时间运行的 Agent 不应缓存旧的 `available_actions`。

## `next` 的地位

`next --role` 提供旧式的未认证角色投影。它可以用于诊断，但无法代表某份 credential 的真实 scope。

Agent 自动化使用 `bootstrap`。角色策略、grant 和资源限制都通过该入口合并。

## 推荐循环

```text
bootstrap
  |
  v
read context_requests
  |
  v
select available_action
  |
  v
execute with state_revision
  |
  v
bootstrap again
```

CLI 没有返回可推进动作时，Agent 读取当前 context 和错误信息，必要时将阻塞原因交给 Master。
