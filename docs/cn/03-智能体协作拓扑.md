# 推荐的智能体协作拓扑

中文 | [English](../en/03-agent-topology.md)

CHASSISS 以 actor 和 role 表示身份。actor 标识长期主体，role 限定该主体当前使用的能力。一个 Agent 可以持有多份 credential，但每条命令只使用其中一份。

## 最小拓扑

推荐配置包含两个常驻 Agent 和一个按需 Designer Session。

```text
Master <-> Designer Session
              |
              v
Build Agent ------> Review Agent
O + D               Reviewer
```

Build Agent 持有 Orchestrator 与 Developer 两份 credential。两份 credential 使用相同 actor。Review Agent 使用不同 actor 的 Reviewer credential。

Designer Session 与执行 Session 分离。它可以由单独 Agent 承担，也可以由 Build Agent 在实施前另开 Session。设计上下文不与实施上下文混用。

## 角色边界

### Master

Master 使用 Master Root。

- 签发和回收角色 credential
- 接受或拒绝 Artifact
- 接受 Mission 验收
- 取消无法继续的 Task
- 在需要时执行 publish
- 读取 Owner history

Master Root 不能执行 `owner apply`。人类独立维护使用单独的 Owner credential。

### Designer

Designer 负责 Requirements、Architecture、Mission 和 Task。

- 从 CLI 获取模板
- 保持 digest 与依赖关系一致
- 定义 Task 范围、预算与 checks
- 根据 Build 或 Review Agent 提供的候选 diff 做设计一致性复核
- 处理 Master 的 Artifact rejection
- 在预算、范围或架构变化时提供 replacement Task

Designer 不能接受自己提交的 Artifact。

### Orchestrator

Orchestrator 负责执行阶段调度。

- 激活、阻塞和恢复 Mission
- 分派、领取、阻塞、恢复、释放和 supersede Task
- 在所有 Task 关闭后提交 Mission 验收
- 发布正式 baseline

Orchestrator 不能批准 Developer submission。

### Developer

Developer 只处理分派给自身 actor 的 Task。

- 打开受控 worktree
- 读取 Task contract 和 change request
- 查看 diff 与状态
- 运行 checks
- 记录 checkpoint
- 提交 immutable submission
- 在无法安全继续时阻塞工作

Developer 不能修改 Task 允许路径之外的文件。

### Reviewer

Reviewer 处理由其他 actor 产生的 submission。

- 读取 submission、Task、文件与 diff
- 运行机械检查
- 记录 approve 或 request_changes
- 对批准的 submission 执行集成

Reviewer actor 与 submission actor 必须不同。

### Owner

Owner 用于人类独立维护。

- 只在项目静止期工作
- 只接管普通项目文件
- 由 CLI 创建一个正式提交
- 产生签名审计记录

同一项目最多保留一个未回收 Owner grant。

## Build Agent 的双角色配置

Build Agent 使用同一个 actor 签发两份 credential。

```text
auth issue --actor build-1 --role orchestrator \
  --output ~/.chassiss/cred-build-orchestrator.yaml
auth issue --actor build-1 --role developer \
  --output ~/.chassiss/cred-build-developer.yaml
```

Orchestrator 可以执行 `task claim`，前提是相同 actor 仍有有效 Developer grant。也可以显式执行 `task assign M001-T001 --owner build-1`。

每次切换职责时显式选择 credential。这样事件会记录准确 role，CLI 也会按对应策略返回能力。

## 多 Developer 配置

大型 Mission 可以签发多个 Developer actor。

```text
developer-a
developer-b
developer-c
```

并行执行仍受以下条件限制。

- 项目 WIP 未达到上限
- Task 依赖已经满足
- `allowed_paths` 与 active Task 不重叠
- 每个 Task 使用独立 worktree
- Reviewer 与提交者 actor 不同

新项目默认 WIP 为 2。当前 CLI 没有公开修改 WIP 的初始化参数。

## Session 启动信息

每个 Session 至少需要以下输入。

- 项目路径
- 角色说明
- 对应 credential armor
- 已安装的 CHASSISS Skill

Agent 导入 credential 后运行 `bootstrap`。后续动作从 CLI 获取，Master 无需复制整套角色规则到 prompt。

## 实验性 Master Agent

能力较强的 Agent 可以代行人类 Master，签发 credential 并创建子 Agent。该拓扑适合流程测试和自动化实验。

Master Root 与角色 credential 通常会进入同一信任域。秘密隔离随之失效。项目质量仍取决于模型能力、上下文管理、测试与人工抽查。

实验结果应保留模型版本、prompt、Session 拓扑、Task 数量、失败点和恢复过程。提交 Issue 时附上这些信息，便于复现。

## 调度建议

长期运行的 Build Agent 和 Review Agent 可以由 cron 或 Agent automation 定时唤醒。唤醒动作只负责触发 Session。Agent 仍需先运行 `bootstrap`。

```text
*/5 * * * * cd /path/to/project && git fetch --prune && /path/to/wake-build
*/5 * * * * cd /path/to/project && git fetch --prune && /path/to/wake-review
```

`git fetch` 只更新远端引用。工作流状态继续以同一个受控项目实例中的 `.chassis/` 为准。
