## 已知问题

### 多机器控制状态不自动同步

复现条件为在两台机器分别 clone 同一 Git 仓库并初始化或复制不同时间点的 `.chassis`。

影响包括 Trust revision、事件链、Task ownership 和 operation journal 不能形成单一事实源。两个副本并行写入可能产生无法自动合并的控制历史。

当前处理方式是维护一个权威控制端。Git remote 只用于正式代码 baseline。

状态为已知架构限制。

### Windows 写操作不可用

Windows 缺少当前 lock backend，写操作返回 `CHS-LOCK-UNSUPPORTED`。

读取和其他行为尚未形成受支持平台基线。当前使用 macOS 或 Linux。

状态为未实现。

### credential 默认长期有效

默认角色 credential 没有 expires_at。credential 泄漏后的暴露窗口持续到 Master revoke。

当前处理方式是签发时设置 TTL、收窄 scope，并保持定期轮换。

状态为安全体验待改进。

### `--persistent` 的输入语义不够严格

credential 在没有到期选项时已经长期有效。当前 CLI 接受 `--persistent`，但没有拒绝它与 `--expires-at` 或 `--ttl-seconds` 同时出现，最终仍按到期时间签发。

当前处理方式是把 `--persistent` 只用于没有任何到期参数的命令。

状态为参数校验待改进。

### credential 文件未使用系统钥匙串

私钥保存在权限为 `0600` 的 YAML 文件中。共享系统用户、恶意进程和不安全备份仍可能读取。

当前处理方式是使用独立系统账户或隔离 runner，并保护 credential 目录。

状态为存储隔离待改进。

### Checks 没有操作系统级 sandbox

结构化 argv、受限环境和 cwd 边界减少了输入面，但 Check 程序仍以当前系统用户权限运行。

当前处理方式是在容器或专用 runner 中运行高风险项目。

状态为执行隔离待改进。

### `review check` 不重新执行 Check 命令

Reviewer 阶段会验证 Submission digest、Git range、scope、budget，以及 Check 结果与精确 snapshot 和 verification source 的绑定。它不会再次启动 Check 进程。

当前处理方式是由 Reviewer 完成语义复核，必要时手工运行项目测试。`integrate apply` 会在 merged candidate 上重新执行冻结 Checks，失败时不会推进 formal baseline。

状态为当前流程设计，后续可以评估是否增加带签名证据的 Reviewer rerun。

### 部分事件文本没有统一大小上限

Review report 和 Owner reason 的上限为 64 KiB。checkpoint、handoff、Mission evidence 和部分 block reason 当前没有同等的统一限制，大文件可能扩大事件和 State。

当前处理方式是只提交简短文本，详细日志保存为项目文件并在证据中引用路径或 digest。

状态为输入限制待统一。

### Git 是唯一 content backend

Config 已记录 `content_backend`，当前只实现 `local-git`。项目需要 Git 根目录和 commit identity。

当前处理方式是使用本地 Git，并按需连接任意 Git remote。

状态为扩展点尚未实现。

### 旧 schema 没有迁移器

旧 Config、State 或 Event 由当前 CLI 明确拒绝。

当前处理方式是保留兼容 CLI，或在独立验证后创建新的 API V2 控制端。

状态为升级工具缺失。

### Publish target 共用 Git adapter

`github`、`gitlab` 和 `remote-git` 当前只区分记录中的 target。平台 API、保护规则、PR 或 release 没有专用集成。

当前处理方式是配置普通 Git remote，并在平台侧独立管理保护规则。

状态为平台能力待扩展。

### CLI 仍是开发版本

版本为 `0.3.0-dev`。协议已有版本和 golden tests，但 release 签名、稳定支持周期与迁移承诺尚未定义。

当前处理方式是固定源码 commit、验证 SHA256SUMS，并让全部 Agent 使用相同 bundled CLI。

状态为发布工程待完善。

### 测试平台覆盖有限

本次完整测试运行于 macOS arm64。仓库包含 Linux amd64、Linux arm64 和 macOS amd64 二进制，但本次没有在对应原生环境运行完整端到端测试。

当前处理方式是验证 bundled checksum，并在目标平台部署前运行标准测试和 smoke test。

状态为 CI matrix 待建立。

## 改进优先级

### P0 控制状态同步

目标是让多机器 Agent 使用同一事件、Trust、operation 和 CAS 后端。

需要定义以下机制。

- 远端签名事件存储
- 单调 Trust revision
- 分布式 operation ownership
- 离线分叉检测
- 控制端迁移协议
- 服务故障下的恢复证据

### P0 credential 隔离

优先探索短期 credential、单动作 token 和 credential broker。Agent 只在执行时取得最小授权，Root 保留在人类或硬件边界内。

### P1 Check sandbox

为 Check runner 增加文件系统、网络、进程和资源限制。macOS、Linux 容器和 CI runner 需要统一的证据模型。

### P1 schema 迁移

提供离线迁移命令，输入旧控制端，输出新目录和迁移报告。迁移过程需要保留源数据、校验所有 digest，并支持失败回滚。

### P1 release 与 CI

建立以下发布门槛。

- macOS 与 Linux 双架构测试矩阵
- Race Detector
- bundled binary smoke tests
- checksum 一致性
- release 签名
- SBOM
- 可复现构建记录

### P2 平台发布 adapter

在 Git fast-forward 证据之外，增加 GitHub 和 GitLab 的保护规则检查、身份确认和可选 release 记录。

### P2 使用体验

可以增加以下能力。

- `chassiss completion`
- 安全的 cache 清理
- journal 只读检查命令
- Task 拆分辅助
- 更紧凑的 bootstrap 输出
- credential 到期提醒
- Issue 诊断包的自动脱敏

### P2 性能与可维护性

大型仓库需要测量 event replay、Git diff、verification digest 和多 Task 列表的性能。后续可以增加可信 snapshot cache，但 cache 命中仍需绑定事件链和 Git identity。
