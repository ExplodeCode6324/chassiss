# CLI 命令参考

中文 | [English](../en/17-cli-reference.md)

本章按生命周期列出 CHASSISS 0.3 命令。Agent 应优先使用 `bootstrap --json` 返回的 `capabilities` 和 `available_actions`。这里的示例用于人工查阅。

## 调用形式

```text
chassiss [global-options] <group> <action> [arguments]
```

全局参数必须位于命令组之前。

| 参数 | 用途 |
| --- | --- |
| `--root <path>` | 指定项目根目录 |
| `--credential <file>` | 指定 Root 或角色 credential |
| `--json` | 输出稳定的 Agent 可读 envelope |
| `--dry-run` | 预览标记，当前 mutation 会返回不支持 |
| `--expect-revision <n>` | 对项目 State 执行 CAS |
| `--expect-trust-revision <n>` | 对 Trust store 执行 CAS |

`--root` 省略时，CLI 从当前目录向上查找 `.chassis`。

当前版本还没有 transactional dry-run。对写命令使用 `--dry-run` 会返回 `CHS-DRY-RUN-UNSUPPORTED`。

## 独立命令

| 命令 | 角色 | 用途 |
| --- | --- | --- |
| `help`、`--help`、`-h` | 无需 credential | 显示命令概览 |
| `version`、`--version` | 无需 credential | 显示 CLI 版本 |
| `bootstrap` | 全部角色 | 返回身份、策略、能力和当前动作 |
| `status` | 全部角色 | 读取项目状态摘要 |
| `next --role <role> [--actor <actor>]` | 兼容入口 | 返回未认证的旧式角色投影 |
| `doctor` | 全部角色 | 检查项目完整性和 Git 健康 |
| `verify` | 全部角色 | 验证项目及可选 credential 锚定 |
| `recover` | 全部角色 | 恢复精确匹配的未完成 journal |
| `explain <error-code>` | 全部角色 | 查询稳定错误码说明 |

`next` 不代表实际 credential 的授权结果。Agent 使用 `bootstrap`。

`status`、`next`、`doctor`、`verify`、`recover` 和 `explain` 可以不带 role credential 运行。`verify` 带 credential 时会额外验证 credential anchor。`bootstrap` 必须验证具体身份。

## `auth`

```text
auth master-init [--output <root-file-or-directory>]
auth inspect [credential-path]
auth issue --actor <actor> --role <role> [scope-options]
auth export <credential-path>
auth import --output <credential-path>
auth revoke <credential-id> [--reason <text>]
```

`auth issue` 由 Master 执行，支持以下选项。

```text
--master-root <path>
--output <path>
--actions <a,b>
--not-before <RFC3339>
--expires-at <RFC3339>
--ttl-seconds <n>
--projects <id,...>
--missions <id,...>
--tasks <id,...>
--submissions <id,...>
--submission-digests <digest,...>
--heads <commit,...>
--baselines <commit,...>
--persistent
```

可签发角色为 Designer、Orchestrator、Developer、Reviewer 和 Owner。Master 身份来自 Root。

未设置到期选项时，credential 默认长期有效。当前 `--persistent` 只用于显式表达这一意图，不应与 `--expires-at` 或 `--ttl-seconds` 同时使用。

## `project`

```text
project init <path> [--existing]
  [--master-root <path>]
  [--max-changed-files <n>]
  [--max-diff-lines <n>]
  [--max-commits <n>]
```

Greenfield 初始化创建 Git 项目、正式分支、`.gitignore` 和初始 commit。Brownfield 使用 `--existing` 接管干净且已有 commit 的 Git 根目录。

## `template`

Designer 使用内嵌模板。

```text
template list
template get <requirements|architecture|mission|task>
  [--id <id>]
  [--output <project-path>]
```

已有输出文件不会被覆盖。

## `artifact`

```text
artifact check <path>
artifact submit <path>
artifact list [--pending]
artifact context <submission-id>
artifact accept <submission-id>
artifact reject <submission-id> --reason <text>
```

Designer 执行 check、submit 和 list。Master 执行 context、accept、reject 和 list。Reviewer 可以读取 artifact context。

## `mission`

```text
mission list
mission context <mission-id>
mission activate <mission-id>
mission block <mission-id> --reason <text>
mission resume <mission-id>
mission submit-acceptance <mission-id> --evidence <file-or-text>
mission accept <mission-id>
```

Orchestrator 负责激活、阻断、恢复和提交验收。Master 接受 Mission。

## `task`

```text
task list [--ready|--active|--blocked|--review]
task context <task-id>
task claim <task-id>
task assign <task-id> --owner <developer-actor>
task block <task-id> --reason <text>
task resume <task-id>
task release <task-id>
task cancel <task-id> --reason <text>
task supersede <task-id> --replacement <new-task-id>
```

Orchestrator 执行 claim、assign、block、resume、release 和 supersede。`claim` 要求相同 actor 另有 Developer grant。Master 执行 cancel。

## `work`

Developer 使用以下命令。

```text
work open <task-id>
work context <task-id>
work status <task-id>
work diff <task-id>
work check <task-id> (--all|--id <check-id>)
work checkpoint <task-id> --file <checkpoint-file-or-text>
work submit <task-id> --file <handoff-file-or-text>
  [--message <summary>]
work block <task-id> --reason <text>
```

`work check` 必须明确选择 `--all` 或一个 Check ID。

## `review`

```text
review list
review history [--task <task-id>] [--submission <submission-id>]
review context <submission-id>
review check <submission-id>
review approve <submission-id> --report <file-or-text>
review request-changes <submission-id> --report <file-or-text>
```

Reviewer 执行 list、context、check 和 verdict。Developer、Orchestrator、Reviewer 与 Master 可以按各自 scope 读取 history。

## `integrate`

批准 Submission 的同一 Reviewer 执行集成。

```text
integrate check <submission-id>
integrate apply <submission-id>
```

`integrate apply` 重新验证 approval、Head、baseline、Checks 和 merged tree。

## `publish`

Orchestrator 或 Master 使用 Git remote adapter。

```text
publish check --target <github|gitlab|remote-git>
  [--remote <name>]
  [--branch <name>]

publish apply --target <github|gitlab|remote-git>
  [--remote <name>]
  [--branch <name>]
```

默认 remote 为 `origin`，branch 为项目配置的正式分支。

## `owner`

```text
owner apply --reason <text>
owner history
```

Owner 执行 apply 和 history。Master 也可以读取 history。

## JSON envelope

Agent 调用应加 `--json`。成功响应包含 API version、命令结果和 revision 信息。错误响应包含稳定错误码、消息、retryable 标记、remediation 和可选 diagnostic category。

调用方根据字段处理结果，不解析终端颜色或自然语言段落。

## 发现当前可用命令

推荐入口如下。

```sh
chassiss --json --root /path/to/project \
  --credential /path/to/credential \
  bootstrap
```

`capabilities` 列出角色可能使用的命令和参数。`available_actions` 只列出当前 revision 下满足状态条件的候选动作。命令执行时仍会再次验证全部约束。
