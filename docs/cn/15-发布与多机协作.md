# Publish 与多机器协作

中文 | [English](../en/15-publish-and-multi-machine.md)

Integration 推进本地 formal baseline。Publish 将该 baseline 以 fast-forward 方式同步到 Git remote。两项操作分别产生 `integration.applied` 和 `publication.applied` 事件。

当前的 `github`、`gitlab` 和 `remote-git` target 共用 Git remote adapter。target 用于记录发布意图，不会调用平台专有 API。

## 准备远端

项目需要一个可访问的 Git remote。

```sh
git remote add origin <remote-url>
```

CLI 默认使用 `origin` 和项目配置的正式分支。也可以显式指定。

## 发布前检查

Orchestrator 或 Master 运行以下命令。

```sh
chassiss --root /path/to/project \
  --credential ~/.chassiss/credentials/orchestrator.cred \
  publish check --target github
```

指定 remote 与 branch 的形式如下。

```sh
chassiss --root /path/to/project \
  --credential ~/.chassiss/credentials/orchestrator.cred \
  publish check --target remote-git \
  --remote upstream \
  --branch main
```

检查要求正式工作区干净、默认分支 Head 等于 State baseline、remote 名称和 branch 合法、远端 URL 可用。

返回状态包括 `ready`、`up_to_date`、`remote_unknown` 和 `diverged`。

## 执行发布

`ready` 状态可以执行发布。

```sh
chassiss --root /path/to/project \
  --credential ~/.chassiss/credentials/orchestrator.cred \
  publish apply --target github
```

CLI 会获取精确远端 Head，确认它为空或是本地 baseline 的祖先，再推送以下 refspec。

```text
<formal-baseline>:refs/heads/<branch>
```

命令不使用 force。推送完成后，CLI 再次读取远端 Head，确认它等于 journal 中的目标 baseline，然后提交 publication event。

## 非 fast-forward

远端包含本地 baseline 未包含的 commit 时，发布返回 `CHS-PUBLISH-NON-FAST-FORWARD`。

不要使用 `git push --force`。先检查远端提交，将需要保留的变更通过明确的 Task 纳入本地正式历史，再重新执行 publish。

远端 Head 尚未在本地对象库中时，`publish check` 可能返回 `remote_unknown`。`publish apply` 会获取精确远端 commit 后重新判断祖先关系。

## endpoint 绑定

State 和 journal 保存 remote 名称与远端 URL 的 SHA-256 digest，不保存原始 URL。URL 可能包含用户名、token 或其他敏感内容。

未完成 publish 期间，remote URL 发生变化会触发完整性阻断。恢复需要还原 journal 对应的精确 endpoint，或由 Master 检查现场。

## Publish recovery

推送可能已经到达远端，但本地进程在写入事件前退出。此时 `recover` 比较远端 Head。

- 远端仍是推送前 Head 且 journal 处于 `prepared`，操作可以取消
- 远端等于目标 baseline，恢复提交原 publication event
- 远端为其他值，进入 `CHS-INTEGRITY-BLOCKED`

发生 push 错误后，先运行 `recover`，再决定是否重试。

## 定时监听

Agent 可以每五分钟检查一次项目。自动化建议执行只读步骤。

```sh
chassiss --json --root /path/to/project \
  --credential ~/.chassiss/credentials/current.cred \
  bootstrap
```

Build Agent 根据 `available_actions` 领取或推进 Task。Review Agent 检查待复核 Submission。Master 或 Orchestrator 在有新 formal baseline 时运行 publish check。

定时任务不应自动 force push、删除 lock、修改 `.chassis` 或忽略 revision conflict。

## 多机器版本同步

所有 Agent 必须看到相同项目内容版本。GitHub、GitLab、自建 Git remote 或其他能可靠同步 commit 的工具都可以承担内容同步。

推荐过程如下。

1. 权威控制端完成 Integration
2. Orchestrator 或 Master 发布正式 baseline
3. 其他机器获取该精确 commit
4. Agent 验证 Head 和 CLI 版本
5. Agent 再开始只读分析或连接权威控制端执行受控动作

CHASSISS 不要求托管平台，但当前 v0.3 的内容 backend 仍依赖本地 Git。

## 当前控制端限制

Publish 只同步代码 baseline。以下内容不会进入 Git remote。

- `trust.yaml`
- 事件链
- State 投影
- operation journals
- Submission manifests
- worktree 绑定

因此，另一个 clone 只有相同代码，没有相同控制历史，也不能独立承担审批、集成或恢复。

当前部署应维护单一权威 `.chassis`。如果迁移控制端，需要在所有写操作停止后同步完整控制目录和 Git objects，并进行完整校验。两个副本不能同时写入。

## 后续方向

多控制端协作需要一个能同步签名事件、Trust revision、operation ownership 和 CAS 的 backend。该能力还需要处理离线分叉、远端锁租约和恢复证据，目前未实现。
