# 安全模型与已知风险

中文 | [English](../en/19-security-model.md)

CHASSISS 面向遵守项目流程的 Agent 和人类开发者。CLI 负责权限、状态、范围、证据和恢复检查。系统不能在已被完全控制的操作系统账户中建立强秘密隔离。

## 受保护的资产

- Master Root 私钥
- 角色 credential 私钥
- Root 签名的 Trust store
- 签名事件链
- 正式 baseline
- Artifact 和 Task 契约
- Submission、review 与 integration 证据
- operation journals

## 信任基础

项目 config 记录 Root fingerprint。Trust store 包含 Root 公钥、grants、revocations 和 Root 签名。角色 credential 中的私钥需要与当前 grant 公钥匹配。

每个写操作同时验证以下条件。

- credential 属于当前项目
- Root fingerprint 一致
- grant 存在且未回收
- 时间窗口有效
- role policy 允许 action
- resource scope 覆盖目标
- 当前 State 允许状态转换
- revision 和 Git 证据仍然匹配

## 能够限制的行为

在 Agent 通过受信 CLI 工作的前提下，CHASSISS 可以阻止以下行为。

- 自报角色或扩大 action
- 使用已回收或过期 credential
- Developer 修改 Task 范围外文件
- Developer 修改独立验证源
- 同一 actor 自写自批
- 未经批准集成代码
- 在未验证 merged tree 时推进 baseline
- 非 fast-forward 发布
- 人工 Owner 修改控制数据和 Artifact
- 在无法确定现场时自动恢复

## 无法提供的隔离

多个 Agent 共用同一操作系统用户时，任何能读取文件系统的 Agent 都可能访问其他 credential、Git 配置、shell history、剪贴板或聊天记录。

在这种部署中，角色分离主要提供工作流约束和审计记录。它不构成针对恶意 Agent 的秘密隔离。

需要更强隔离时，为角色使用不同系统账户、容器或远程执行环境，并只暴露各自的 credential 和工作目录。

## 长效 credential

默认角色 credential 没有过期时间，便于持续运行的 Agent 使用。长期私钥增加泄漏窗口。

生产项目建议设置 `--ttl-seconds` 或 `--expires-at`，并使用最小 resource scope 和 action scope。周期性自动化可以在到期前由 Master 签发新 credential，再回收旧 credential。

当前版本没有在线 credential broker，也没有一次性 action token。

## Base64 armor

Armor 使 credential 适合文本传输，并提供 envelope version 和 payload digest。它不隐藏私钥。

避免通过公开 Issue、普通日志、录屏、长期共享聊天或不可信剪贴板服务分发 armor。泄漏后立即 revoke 对应 credential，并检查该 grant 在泄漏窗口内签名的事件。

## Root 风险

Master Root 能签发角色授权并作出 Master 决策。Root 泄漏可能破坏整个项目的身份边界。

建议措施如下。

- 存放在项目目录之外
- 限制为人类可访问的账户
- 使用加密磁盘和可靠备份
- 不发送给 Agent Session
- 不通过 `auth export` 传输
- 不复制到 CI 日志或环境快照

当前 Root 文件自身没有口令加密或硬件密钥支持。

## Trust rollback

Trust store 位于本地控制端。攻击者若同时替换 Trust、config 和相关控制状态，可能尝试回滚授权视图。Root signature、fingerprint 和 revision 能发现普通篡改，但当前没有外部透明日志或远端单调计数器。

不要用备份中的旧 `trust.yaml` 覆盖现有文件。控制端备份应包含完整 `.chassis` 和时间点说明。

## CLI 完整性

权限检查依赖正在执行的 CLI。被替换的二进制可以跳过本地规则或泄漏 credential。

使用仓库提供的 `SHA256SUMS` 验证 bundled CLI。所有 Agent 应使用同一发布版本。更高保障的部署可以增加签名 release、只读安装目录和可复现构建验证。

## Git 与外部工具

Agent 若绕过 CLI 直接修改正式分支、worktree refs 或 remote，会破坏 CHASSISS 的证据绑定。`verify` 能发现多类偏离，但无法撤销已经泄漏的代码或秘密。

限制 Agent 可使用的系统工具和远端 token。Publish credential 应只允许目标仓库和必要 branch。

## Check 执行风险

Checks 执行项目内声明的命令。恶意或被污染的 Artifact 可能运行危险程序。默认无 shell 和受限环境减少输入面，但当前版本没有操作系统级 sandbox。

Master 接受 Task 前应审查 CheckSpec。高风险项目在容器或隔离 runner 中执行 CHASSISS。

## 聊天与日志

Agent 上下文、终端历史和错误报告可能携带路径、源码或秘密。`--json` 输出不应包含 credential 私钥，但外部命令的 Check 输出由项目工具产生。

提交 Issue 前清理 token、remote URL、credential armor、Root 和私有代码。

## 实验性 Master Agent

能力足够强的 Agent 可以代行人类 Master，分发子 Agent 并推进整个项目。若这些 Agent 共享同一执行环境，它们可能同时接触 Root 和角色 credential，默认秘密隔离随之失效。

该模式可以测试流程自动化和模型能力，不保证项目质量。建议只用于可公开、可恢复的实验项目，并详细记录模型、prompt、失败点和人工介入。

## 改进方向

- 短期和单动作 credential
- 系统钥匙串与硬件密钥
- 独立 credential broker
- 签名 release 与可复现构建
- Check sandbox
- 远端透明事件日志
- 多控制端一致性协议
- credential 使用告警与审计导出
